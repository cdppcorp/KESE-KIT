# Constitution — 집필 규칙

> KESE KIT: Korea Enhanced Security Evaluation - KISA Infrastructure Toolkit
> 도서 분류: 실용/교육 가이드
> 대상 독자: 바이브코딩을 하는사람, 개발자지만 서버를 운용하는 사람, 정부과제로 개발을 하는사람, 공공조달시장에서 개발을 하는 사람, 정부관련 코딩하는 사람, 서버를 처음 다루는 사람

---

## 1. 문체 규칙

### 1.1 문장 종결어미
- **기본 어미**: 합니다체 (예: ~합니다, ~됩니다, ~입니다)
- 명령/지시문: ~하세요, ~하십시오
- 설명문: ~합니다, ~입니다

### 1.2 경어 수준
- **기본**: 평어 (독자와 동등한 위치에서 설명)
- 기술 설명 시 친근하면서도 전문적인 톤 유지

### 1.3 문장 스타일
- 한 문장은 가능한 60자 이내로 유지
- 복문보다 단문 선호
- 능동태 우선 사용

---

## 2. 용어 표기 규칙

### 2.1 기본 형식
```
한글(영문, 약어)
```

예시:
- 주요정보통신기반시설(Critical Information Infrastructure, CII)
- 취약점 분석·평가(Vulnerability Analysis and Assessment, VAA)
- 한국인터넷진흥원(Korea Internet & Security Agency, KISA)

### 2.2 용어 일관성
- 첫 등장 시 전체 형식 사용
- 이후 등장 시 한글 또는 약어 사용 가능
- glossary.md에 등록된 용어 우선 사용

### 2.3 영문 표기
- 고유명사: 원어 그대로 (예: KISA, ISMS-P)
- 일반 기술용어: 한글 번역 우선, 괄호 안에 영문

---

## 3. 특수 요소

### 3.1 팁 박스
```markdown
> **TIP**
> 팁 내용을 여기에 작성합니다.
```

용도: 유용한 정보, 권장 사항, 시간 절약 방법

### 3.2 주의 박스
```markdown
> **WARNING**
> 주의 내용을 여기에 작성합니다.
```

용도: 보안 위험, 흔한 실수, 주의해야 할 사항

### 3.3 코드 블록
```markdown
```bash
# 명령어 설명
command --option value
```
```

- 모든 코드/명령어는 코드 블록으로 표시
- 언어 지정 필수 (bash, python, yaml 등)
- 주석으로 간단한 설명 포함

---

## 4. 상호참조 규칙

### 4.1 내부 참조
- 장 참조: "N장 참조" 또는 "N장에서 설명합니다"
- 절 참조: "N-M절 참조" 또는 "N-M절에서 다룹니다"

### 4.2 외부 참조
- 공식 문서: "KISA 가이드 참조"
- 법령: "정보통신기반보호법 제N조"

---

## 5. 구조 규칙

### 5.1 장/절 구성
- 각 장은 도입부로 시작 (해당 장에서 다룰 내용 요약)
- 각 절은 명확한 목적을 가짐
- 장 끝에는 요약 또는 체크리스트 포함 권장

### 5.2 그림 및 표
- 그림: `[그림 N-M] 제목` 형식
- 표: `[표 N-M] 제목` 형식
- 모든 그림/표는 본문에서 언급 필수

---

## 6. 대상 독자 고려사항

### 6.1 기술 수준 가정
- 기본 리눅스 명령어 이해
- 서버 운영 기초 지식
- 보안 개념 기초 이해 (취약점, 패치 등)
- JavaScript/Python 기본 문법 이해 (시큐어코딩 파트)

### 6.2 설명 수준
- 정부/공공 환경 특수성 설명 포함
- 실무 적용 가능한 구체적 예시 제공
- 바이브코딩 환경에서의 활용 방법 포함
- AI 도구(Claude, Cursor, Copilot) 활용 시 보안 유의사항

## 7. 시큐어코딩 항목 작성 규칙

### 7.1 항목 구조 (ref-011, ref-012 기반)
각 보안약점 항목은 다음 순서로 작성합니다:

```markdown
### N. 보안약점명(영문명, CWE-XXX)

**개요**: 취약점 설명 (1~2문장)

**영향**: 발생 시 위험성

> **WARNING**
> 이 취약점이 발생하는 대표적인 상황 설명

**안전하지 않은 코드**
```언어
// 취약한 코드 예시
```

**안전한 코드**
```언어
// 안전한 코드 예시
```

> **TIP**
> 점검 기준 및 추가 권장사항
```

### 7.2 코드 예시 규칙
- 안전하지 않은 코드와 안전한 코드는 반드시 쌍으로 제공
- 코드 블록에 언어 지정 필수 (javascript, python, sql, bash 등)
- 주석으로 취약 지점 / 수정 지점 표시
- 실무 프레임워크 기반 예시 우선 (Express.js, Django, Flask 등)

### 7.3 CWE 표기 규칙
- 항목 제목에 CWE 코드 병기: `SQL 삽입(SQL Injection, CWE-89)`
- CWE 코드는 약어로 직접 사용 가능 (예: CWE-79)
- 동일 CWE가 JS/Python 양쪽에 있을 경우 각각의 언어 특성 반영
